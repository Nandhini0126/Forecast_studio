Random Forest Model
We use a Random Forest Regressor with 400 estimators (decision trees). Each tree learns a slightly different view of the data through bootstrap sampling and random feature subsets. The ensemble's averaged predictions are more robust and less prone to overfitting than any single tree. The model excels at capturing non-linear relationships between calendar features, lagged sales, and rolling statistics.
Why 28-Day Horizon
The 28-day (4-week) forecast horizon aligns with typical retail replenishment cycles. It's long enough to inform purchasing and staffing decisions while short enough to maintain high accuracy. A 4-week window also captures full weekly seasonality patterns — every day of the week appears exactly 4 times, providing balanced predictions.
Data Leakage Prevention
To prevent data leakage, all lag features start at lag ≥ 28 (matching the forecast horizon). This ensures the model never "sees" future sales during training. We also use a strict time-based train/test split — the test set always comes chronologically after the training set, simulating real-world deployment where you only have past data.
Baselines Matter
We compare against two baselines: Weekly Naive (repeat last week's sales) and Yearly Naive (use same period last year). If our model can't beat these simple methods, it's not adding value. The best baseline's MAE is shown alongside model MAE so you can immediately gauge the improvement — a true "sanity check" for model performance.
Feature Engineering
The model uses three categories of engineered features: Calendar features (day-of-week, month, quarter, year, holidays, weekend flag), Lag features (sales at t−28, t−35, t−42, t−56, t−91 days), and Rolling statistics (28-day and 56-day rolling mean, std, min, max). These features capture both short-term momentum and long-term seasonal patterns in the data.
